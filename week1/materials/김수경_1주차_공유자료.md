# 1. MCP의 등장 배경

## 1-1. LLM의 한계

- 대규모 언어 모델(LLM: Large Language Model)은
    - 학습 시점의 지식에 고착화되어 있고
    → 실시간 데이터에 액세스할 수 없다: `오늘 날씨 알려줘 (X)`
    - 외부 세계와 상호작용할 수 없다.
    → `회의실 예약해줘 (X)`

## 1-2. RAG(Retrieval-Augmented generation): LLM의 지식을 확장

- 문서를 데이터 베이스에 저장해놓고 → 질문에 답변할 수 있는 문서를 검색하여 → 그 문서를 LLM에게 전달
- LLM이 새로운 지식(문서)를 기반으로 답변할 수 있도록 함

# 0. 다양한 엔지니어링 기법들

- System Prompt
    - AI 모델의 동작 방식을 제어하는 설정값으로, 대화의 스타일, 성격, 제약 조건 등을 결정하는 역할
        
        ```jsx
        response = openai.ChatCompletion.create(
            model="gpt-4",
            messages=[
                {"role": "**system**", "content": "너는 유머러스하고 장난기 많은 AI야. 모든 답변을 재미있게 해줘."},
                {"role": "**user**", "content": "컴퓨터가 왜 느릴까?"}
            ]
        )
        
        print(response["choices"][0]["message"]["content"])
        ```
        
    - 사용자 프롬프트: 사용자가 AI 에게 요청할 작업 및 질문
    
    ---
    
    https://github.com/asgeirtj/system_prompts_leaks/blob/main/Anthropic/claude-opus-4.6.md
    
    - 자기 정체성(출처는 Anthropic이다), 지식의 컷오프(언제까지의 정보만 알고 있다), 행동 강력ㅇ(친절하고 명확하게 답해라), 금기사항( 묻지 않으면 특정 주제를 언급하지 마라)
- Prompt Engineering: https://cloud.google.com/discover/what-is-prompt-engineering?hl=ko
    - LLM이 원하는 응답을 생성할 수 있도록 프롬프트를 설계하고 최적화하는 기술
    - 실사형 이미지
        
        
        | 안내 | 프롬프트 예시 |
        | --- | --- |
        | 객체, 풍경, 조명, 스타일 등 원하는 이미지를 자세히 설명하는 프롬프트를 작성하여 사실적인 고품질 이미지를 생성합니다. | "바다 위 해 질 녘 하늘을 배경으로 야자수 실루엣이 보이는 실사 같은 이미지" |
    - 모범사례: https://aws.amazon.com/ko/what-is/prompt-engineering/#ams#what-isc6#pattern-data
- Context engineering
    - 사용자가 어떤 환경에서 AI를 활용하는지, 어떤 목적과 맥락을 가지고 있으며, 어떤 정보를 어떤 방식으로 제공받기를 기대하는지를 이해하고 이에 맞춰 설계된 컨텍스트를 AI 모델에게 제공해서
    → 어떤 기준과 맥락 위에서 문제를 해석하도록 할 것인가를 설계하는 영역
    - 프롬프트 엔지니어링이 ‘무엇을 물어볼 것인가’, ‘어떻게 질문할 것인가’에 초점을 맞춘다면, 컨텍스트 엔지니어링은 프롬프트를 포함해 질문이 해석되는 정보 환경 전체를 설계하는 방식 
    → 프롬프트 엔지니어링은 컨텍스트 엔지니어링의 하위 영역
- agent engineering
    - 목표가 주어지면 에이전트는 요청을 더 작고 관리 가능한 하위 작업으로 나눔. 그런 다음 파일 시스템에 액세스하거나, bash 명령어를 실행하거나, 버전 제어 시스템과 상호작용하는 등의 특정 도구를 사용하여 이러한 작업을 수행: **스스로 계획을 세우고 도구를 골라서** 실행
    - 반복적인 피드백 루프: 테스트 사례를 작성하고, 코드를 실행하고, 실패를 관찰한 다음, 테스트를 통과하도록 코드를 다시 작성

## 4-2. 자동 강제 시스템

![image.png](%EA%B9%80%EC%88%98%EA%B2%BD_1%EC%A3%BC%EC%B0%A8_%EA%B3%B5%EC%9C%A0%EC%9E%90%EB%A3%8C/image%2021.png)

- 린터: 코드의 맞춤법 검사기
- 프리커밋 훅: 코드를 저장하기 직전에 자동으로 돌아가는 검사 스크립트
- 자동 교정 루프: 린터 빨간불 → 에이전트가 스스로 수정: 사람 개입 불필요

## 4-3. 가비지 컬렉션

![image.png](%EA%B9%80%EC%88%98%EA%B2%BD_1%EC%A3%BC%EC%B0%A8_%EA%B3%B5%EC%9C%A0%EC%9E%90%EB%A3%8C/image%2022.png)

- AI가 만들어놓은 안좋은 코드를 주기적으로 자동 청소하는 것
    - 기존 크드의 나쁜 코드를 그대로 따라하기 때문에
    - 문서가 실제 코드와 달라진건 없는지
    - 규칙을 위반한 코드가 생겼는지
    - 사용하지 않는 코드가 쌓였는지를 자동으로 찾아서 수정
