하네스(Harness)란?

Agent = Model  +  Harness
모델이 아무리 똑똑해도 하네스(주변 시스템) 설계가 에이전트 성능을 결정한다

🏎️ 레이싱카 비유

드라이버(Model) — 아무리 실력이 좋아도

레이싱카(Harness) — 차가 구식이면 못 이긴다

F1에서 드라이버 실력은 10%, 차의 성능이 90%라는 말이 있다

🍳 주방 비유

프롬프트 = "짜장면 하나요" (What)

하네스 = "면 7분, 춘장 3분, 맛 검증까지" (How)

무엇을 원하는지 말하는 건 쉽다 — 어떻게 도달할지를 설계하는 게 하네스

6개 레이어로 구성

CLAUDE.md → Hooks → Skills → Memory → MCP → Subagents

하네스 = 모델이 아닌 모든 것 — 코드, 설정, 규칙, 도구, 기억, 실행 로직의 총체

핵심 공식부터 볼게요. Agent = Model + Harness. 이게 오늘의 전부예요. 레이싱카에 비유하면, 드라이버가 모델이고 차가 하네스예요. F1에서 드라이버 실력은 전체 성능의 10%, 나머지 90%는 차의 설계와 팀 전략이라는 말이 있거든요. AI도 마찬가지예요. GPT-4가 얼마나 똑똑한지보다, 그 주변에 뭘 깔아줬느냐가 실제 성능을 결정해요. 주방 비유로도 보면, "짜장면 하나요"는 프롬프트예요. 그런데 그걸 만들어내는 주방 시스템 전체 — 면 삶는 시간, 춘장 볶는 순서, 완성 후 맛 검증까지 — 이게 하네스예요. 하네스는 6개 레이어로 구성되는데요, 지금 당장 다 외울 필요 없고, 이따가 하나하나 우리 실제 도구와 연결해드릴게요.

모델 혼자서는 못하는 것 4가지

단카톡 / 계산기 비유

🗂️ 상태 유지 — "이미 잊어버렸어요"

단체 카톡방에 공지를 올렸는데 메시지 1000개가 쌓이면? 공지가 묻혀버린다

→ 하네스가 기억을 저장/복원해줌 (Obsidian, Memory)

🖩 코드 실행 — "공식은 알아도 계산기는 없어요"

수학 공식을 외워도 큰 수 계산은 계산기가 필요하다. AI도 마찬가지

→ 하네스가 실행 환경 제공 (파일시스템, 셸)

🔍 실시간 지식 — "어제 외운 정보예요"

시험 전날 밤새 외운 내용 vs 오픈북 시험 — 검색 가능한 상태가 훨씬 낫다

→ 하네스가 도구 연결 (MCP: Jira, Context7)

⚙️ 환경 설정 — "재료 없이 레시피만 알아요"

요리 방법을 알아도 재료와 도구가 없으면 만들 수 없다

→ 하네스가 패키지 설치, 샌드박스 구성

모델은 "두뇌"일 뿐 — 손발, 기억, 도구, 환경을 붙여주는 게 하네스의 역할

모델이 아무리 똑똑해도 혼자서 못하는 것이 딱 4가지 있어요. 첫째, 기억 유지. 단카톡 공지 비유가 딱 맞아요. 처음엔 잘 보이는데 메시지가 1000개 쌓이면 중요한 공지가 묻혀버리죠. AI도 대화가 길어지면 앞의 중요한 지시를 "잊어버려요". 이걸 Context Rot, 컨텍스트 부패라고 불러요. 둘째, 코드 실행. AI는 코드를 쓸 줄 알아도 실행은 못 해요. 수학 공식을 아는 것과 계산기를 가진 것은 다르죠. 셋째, 실시간 지식. AI의 지식은 학습 시점에 멈춰있어요. 오픈북 시험처럼 검색할 수 있게 해주는 게 MCP예요. 넷째, 환경 설정. "패키지 설치해줘"를 말할 수는 있는데 실제로 설치하려면 환경이 있어야 해요. 이 4가지를 모두 해결해주는 게 하네스입니다.

LangChain이 Terminal Bench 2.0에서 실증 — 모델 선택 못지않게 하네스 설계가 중요하다

자, 이게 추상적인 이야기가 아니라는 증거를 보여드릴게요. LangChain이 Terminal Bench 2.0이라는 벤치마크에서 실험했어요. 모델은 그대로 두고 하네스만 손봤더니 52.8%에서 66.5%로 올랐어요. 13.7%p가 올랐는데, 이게 얼마나 큰 건지 맥락을 드리면, 보통 신형 모델 한 세대가 올라가면 10~15%p 오르거든요. 즉, 하네스 최적화 하나가 모델 세대 교체와 맞먹는 효과를 냈어요. 축구팀 비유로 보면, 선수단(모델)은 그대로인데 코치진(하네스)을 바꿨더니 성적이 확 올랐다는 거예요. 이게 오늘 하네스 이야기를 하는 이유입니다.

"스킬이 없다"가 아니라 "스킬 간 연결이 없다"가 핵심 문제

지금 상태를 솔직히 보면, 냉장고에 재료는 엄청 많은데 레시피가 없는 주방이에요. 양파도 있고, 고기도 있고, 소스도 있는데, 매번 "양파 어디 있더라?" 하면서 찾고, 볶고 나서 "이제 고기 꺼내야지" 하면서 수동으로 연결하는 거예요. 문제는 "재료가 없다"가 아니라 "레시피가 없다", 즉 오케스트레이션이 없다는 거죠. 그래서 필요한 건 주방장입니다.

좀 더 구체적으로 볼게요. Block 패턴 — 세션 시작할 때 Hook으로 MCP 서버가 자동 시작되죠. LLM이 아무리 "MCP 없이 해볼게요" 해도 막혀요. Evaluate 패턴 — 테스트 파이프라인에서 시나리오 분석가, 테스트 작성자, 리뷰어가 역할이 나눠져 있잖아요. 생성자와 평가자가 분리된 거예요. Context 패턴 — KB 워크플로우에서 작업 전에 KB 신뢰도를 평가하고, ToolSearch로 필요한 도구만 로드하고, Obsidian으로 컨텍스트를 관리해요. Reasoning Sandwich — Opus로 계획하고 Sonnet으로 구현하는 거 이미 하고 있죠.

다음 단계는 "주방장을 고용하는 것"이에요. 재료와 레시피는 있으니, 주방장이 "오늘 메뉴는 이거, 이 순서로 조리해, 맛 검증은 저쪽 셰프가 해" 라고 지휘하는 거예요. 첫 번째는 dev-flow — Jira 티켓 들어오면 분석부터 PR까지 알아서 해주는 오케스트레이터예요. 두 번째는 daily-copilot — 아침엔 브리핑, 저녁엔 일지를 자동으로. 세 번째는 weekly-orchestrator — 금요일에 주간 보고를 알아서 만들어주는 거죠. Block 패턴도 강화할 건데요, PR 만들기 전에 KB 신뢰도를 Hook으로 강제 체크하고, Doom Loop 감지도 추가해서 같은 패턴을 반복하면 자동으로 멈추게 할 거예요.

dev-flow 실제 시나리오

SPPRODUCT-87001 "상품 상세 GNB 개선" 티켓이 들어왔다면?

① Jira MCP — 티켓 자동 분석 [Context 패턴] 티켓 설명 + 댓글 + 첨부파일 읽기 → 요구사항 구조화
② KB 신뢰도 체크 [Context 패턴] GNB KB 65점/75점 → ⚠️ 코드 분석 먼저 권장 (감 대신 자료로)
③ planning-with-files [Suggest 패턴] 영향 범위 분석 → task_plan.md 자동 생성 (수정 파일 3개, 테스트 2개)
④ 구현 + unit-test [Loop + Evaluate + Reasoning Sandwich] Opus 계획 → Sonnet 구현 → test-writer → test-reviewer 검증 (통과까지 반복)
⑤ pr-generator [Block + Context] Hook: KB 신뢰도 재체크 강제 → PR 규칙 자동 적용 → PR 생성 + Jira 상태 변경

5가지 원자 패턴 + Reasoning Sandwich가 한 티켓 안에서 전부 동작

구체적으로 어떻게 돌아가는지 볼게요. 가상의 Jira 티켓 하나가 들어왔다고 해볼게요. "상품 상세 GNB 개선" 티켓이에요. 첫 번째, Jira MCP가 티켓을 자동으로 읽어서 요구사항을 구조화해요. Context 패턴이죠. 두 번째, KB 신뢰도를 체크해요 — GNB 관련 KB가 65점인데 필요 점수가 75점이라 코드 분석을 먼저 돌려요. 세 번째, planning-with-files 스킬이 영향 범위를 분석하고 작업 계획서를 자동으로 만들어요. Suggest 패턴이에요. 네 번째, Reasoning Sandwich가 동작해요. Opus가 계획하고, Sonnet이 구현하고, test-writer와 test-reviewer가 검증하는데, 통과할 때까지 반복해요. Loop와 Evaluate가 동시에. 마지막으로 PR 만들 때 Hook이 KB 신뢰도를 다시 체크해요. 5가지 패턴이 한 티켓 안에서 전부 동작하는 거예요. 이걸 dev-flow 오케스트레이터가 자동으로 엮어주는 게 목표입니다.

유튜브를 볼수록 → 알고리즘이 취향 파악 → 더 좋은 영상 추천 → 더 많이 봄 → 반복

"Eval은 하네스의 학습 데이터다" — ML에서 데이터 품질이 모델 성능을 결정하듯, Eval 품질이 하네스 품질을 결정한다

마지막으로 Eval 플라이휠 이야기를 할게요. 유튜브 알고리즘 알죠? 볼수록 취향을 파악하고, 추천이 좋아지고, 더 보게 되는 선순환이에요. 하네스도 똑같아요. 에이전트를 많이 쓸수록 → 실행 기록(trace)이 쌓이고 → 거기서 실패 패턴을 찾아서 Eval 케이스를 만들고 → 그 Eval로 하네스를 개선하고 → 더 잘 작동하면 더 많이 쓰게 되는 선순환이 생겨요. LangChain에서는 이걸 "Hill Climbing"이라고 불러요. 산 꼭대기를 향해 올라가는 것처럼, Eval 점수를 올리는 방향으로 하네스를 반복해서 개선하는 거예요. 핵심 인사이트는 "Eval은 하네스의 학습 데이터다" — ML에서 데이터 품질이 모델을 결정하듯, 어떤 Eval을 갖고 있느냐가 하네스 품질을 결정해요.
