AI Agent Study Week 1

AI 코딩 워크플로우 소개
Harness, Gstack, BMAD-Method

단순한 프롬프팅을 넘어 실제 프로덕션 소프트웨어를 위한 최신 워크플로우

발표자: 신인호

1. 왜 새로운 방법론이 필요한가?

데모 수준의 '바이브 코딩(Vibe Coding)'에서 프로덕션 레벨로 넘어가기 위한 도전

🧩 컨텍스트 한계

프로젝트가 커지면 AI가 이전 코드를 잊거나 일관성을 잃는 Context Anxiety 발생

⚖️ 자기 평가 한계

AI는 자신의 실수를 발견하기 어렵고, 오류가 있어도 "완벽하다"고 과대평가하는 경향

🗝️ 암묵지 전달 한계

팀 내부의 아키텍처 패턴이나 라이브러리 규칙 등 명시되지 않은 정보를 AI는 알 수 없음

"AI 모델 자체의 지능보다 AI가 일하는 환경을 어떻게 통제(엔지니어링)할 것인가가 핵심입니다."

2. 하네스 엔지니어링 (Harness Engineering)

"AI가 실수하지 않도록 환경(안전벨트)을 설계하라"

핵심 개념

  • 프롬프트를 넘어 역할과 파이프라인을 구조적으로 강제
  • AI가 딴짓을 못 하도록 '안전망' 구축
  • 미첼 하시모토(하시코프 창립자) 제안

Maker vs Checker (분리)

Planner: 스펙 정의

Generator: 코드 구현

Evaluator: 실제 환경 검증

모델이 똑똑해지더라도, 모델의 빈틈을 메워주는 환경 설계는 계속 필요합니다.

3. Gstack: 실행력 중심 스타트업 워크플로우

"거창한 설계보다 빠른 MVP 런칭과 배포(Ship it!)"

접근 방식

클로드 코드를 범용 챗봇이 아닌, 23개의 특정 전문가 역할로 강제 분리

  • skill.md를 활용한 역할 정의
  • Y Combinator CEO 게리 탄의 세팅

주요 명령어 (Slash Commands)

/plan-ceo-review: 제품 방향성 검증

/review: 시니어 엔지니어 관점 검토

/qa: 브라우저 클릭 테스트

/ship: 테스트 및 PR 생성 한 번에

💡 인사이트: 극도로 빠른 실행력, 소규모 팀, 단기 스프린트에 최적화

4. BMAD Method: 정통 애자일의 AI화

"코드 한 줄을 짜기 전에 완벽한 청사진(PRD/아키텍처)부터 만들자"

전담 에이전트 파이프라인

PRD/Arch

초기 세팅
→
PO

작업 샤딩
→
Scrum Master

Epic/Story 생성
→
Dev

모듈 구현
→
Review

자동 승인

엄격한 프로세스

분석 ➔ 계획 ➔ 아키텍처 ➔ 구현 ➔ 테스트 단계를 강제하여 컨텍스트 유지

문서 기반 관리

PRD를 작은 스토리 단위로 쪼개고 각 에이전트가 이를 하나씩 완수

5. Gstack vs BMAD-Method 요약

구분 Gstack BMAD Method
철학 린 스타트업 (빠른 검증/배포) 정통 애자일 (체계적 계획/구현)
주도권 개발자가 명령어로 AI 호출 오케스트레이터가 흐름 리드
템포 매우 빠름 (문서 최소화) 상대적 느림 (초기 설계 집중)
최적 환경 소규모, MVP 런칭 중대형, 복잡한 로직, 유지보수

💡 결론: 우리 팀의 프로젝트 규모와 요구되는 안정성에 따라 선택하거나 혼합 운용

6. 커스텀 하네스 생성기 (RevFactory)

프로젝트에 최적화된 에이전트 팀과 스킬을 생성해주는 메타-스킬

자동 팀 설계

도메인 분석 후 필요한 에이전트(.agents/)와 스킬(.skills/)을 자동 생성
