1. CLAUDE.md — 프로젝트 기억 시스템

세션마다 새로 시작하는 Claude에게
"이 프로젝트가 뭔지, 어떻게 일해야 하는지"를 자동으로 알려주는 파일

📁 파일 우선순위 계층

/etc/claude-code/CLAUDE.md   ← Managed (조직 전체)
~/.claude/CLAUDE.md          ← User (모든 프로젝트)
./CLAUDE.md                  ← Project (팀 공유)
하위 디렉토리/CLAUDE.md        ← 지연 로드

더 구체적인 위치가 높은 우선순위

✏️ 효과적인 작성법

나쁜 예좋은 예
"코드를 잘 포맷""2칸 들여쓰기, Dart fmt"
"변경사항 테스트""커밋 전 flutter test"
"파일 정리""서비스는 lib/services/"

권장 크기: 200줄 이하

2. Hooks — 라이프사이클 자동화

Claude에게 "해줘"라고 부탁하는 것이 아니라
시스템 수준에서 강제 실행하는 자동화

주요 Hook 이벤트

SessionStart 세션 시작·재개 시 실행
PreToolUse 도구 실행 전 — 차단 가능
PostToolUse 도구 실행 후 — 결과 후처리
SubagentStart 서브에이전트 시작 시
UserPromptSubmit 사용자 입력 직후 (Claude 처리 전)

Hook 타입 4가지

// ① Command (가장 일반적)
{ "type": "command", "command": "./validate.sh" }

// ② HTTP (외부 서비스)
{ "type": "http", "url": "http://localhost:8080/hooks" }

// ③ Prompt (LLM 직접 판단)
{ "type": "prompt", "model": "haiku", "prompt": "..." }

// ④ Agent (서브에이전트 검증)
{ "type": "agent", "prompt": "Verify: $ARGUMENTS" }

종료코드 2 = 차단, 0 = 허용, 기타 = 경고 후 계속

2. Hooks — 이 프로젝트의 실제 구현

① SessionStart / SubagentStart — 역할별 컨텍스트 자동 주입

hook_session_start.py: cwd명 또는 subagent_type으로 역할 감지 → General/ 폴더의 최신 기획 문서·히스토리를 Claude 컨텍스트에 주입

에이전트 시작마다 "어디까지 왔는지, 무슨 스펙인지"가 자동 주입 — 매번 수동으로 알려줄 필요 없음

② PreToolUse (Bash) — master 직접 푸시 차단

hook_pre_git.py: git push origin master/main 감지 시 차단(exit 2) · 모든 git 쓰기 전 verify_local.sh 자동 실행 (flutter analyze + test)

Claude가 실수로 master에 직접 푸시하는 것을 시스템이 막음 → PR Flow 강제

③ PostToolUse (Agent) — 에이전트 완료 알림

hook_agent_notify.py: Agent 도구 완료 후 Discord 알림 발송 — "✅ Dev 에이전트 작업 완료"

3. Skills — 확장 가능한 명령 시스템

"Claude에게 어떤 일을 '어떻게' 할지 가르치는 플레이북"
SKILL.md 파일로 정의하는 재사용 가능한 프롬프트 템플릿 · /skill-name 으로 호출

SKILL.md 파일 구조

---
name: explain-code
description: 코드를 유추로 설명할 때 사용
user-invocable: true      # /explain-code 메뉴
disable-model-invocation: false  # 자동 호출
allowed-tools: Read Grep  # 추가 허용 도구
model: haiku              # 모델 재정의
paths: "**/*.dart"        # 조건부 활성화
---

# Claude가 따르는 지침
1. 일상 사물에 비유
2. ASCII 다이어그램
3. 단계별 설명

동적 컨텍스트 주입 (! 구문)

---
name: pr-summary
context: fork
---
## PR 현황
- Diff: !`gh pr diff`
- 댓글: !`gh pr view --comments`
위 정보를 바탕으로 요약해줘.

실시간 git diff, env 값을 Skill에 주입 가능

4. Agents (Sub-agents) — 전문화된 AI 어시스턴트

왜 서브에이전트가 필요한가?

독립 컨텍스트 윈도우

각 에이전트는 독립된 컨텍스트 → 복잡한 작업에서 토큰 한도 문제 없음

병렬 실행

여러 에이전트가 동시에 다른 파일/기능 작업 → 개발 속도 향상

전문화

각 에이전트가 특정 도메인에 집중 → 더 깊은 전문성

에이전트 정의 파일 구조

---
name: dev
description: 핵심 로직 구현 담당.
             TimelineEngine 작업 시 사용.
tools: Read, Write, Edit, Bash, Glob, Grep
model: sonnet
---

6. Git Worktrees — 병렬 세션 격리

하나의 Git 저장소에서 여러 브랜치를 동시에 체크아웃
에이전트마다 독립된 작업 디렉토리 → 서로 간섭 없이 병렬 개발

이 프로젝트의 5개 워크트리

MustGoOutNow/ master 오케스트레이터 (이 대화)
MustGoOutNow_dev/ role/dev
MustGoOutNow_design/ role/design
MustGoOutNow_devops/ role/devops
MustGoOutNow_qa/ role/qa

PR 워크플로우

1 각 워크트리에서 코드 수정
2 flutter analyze && flutter test
3 git push origin role/dev
4 gh pr create --base master
5 Gemini Reviewer 코드 검수
6 오케스트레이터가 gh pr merge
7 hook이 전체 워크트리 자동 동기화

hook_pre_git.py가 role 워크트리에서 master 직접 푸시 자동 차단

7. Plan Mode — 안전한 읽기 전용 분석

파일 수정·명령 실행 없이 분석과 계획만 수행하는 모드
"무엇을 할지 먼저 합의한 다음, 실행한다"

활성화 방법

# CLI 플래그
claude --permission-mode plan

# 세션 중 토글
Shift+Tab → Shift+Tab

# settings.json 기본값
{ "defaultMode": "plan" }

할 수 있는 것 vs 없는 것

가능불가능
파일 읽기파일 쓰기/편집
Grep, Glob 검색Bash 명령 실행
분석 및 계획 제안코드 변경
요구사항 수집PR 생성

이 프로젝트: 신규 기능 설계 시 orchestrator가 Plan Mode로 전체 영향 분석 → 사용자 승인 후 실행

8. 전체 아키텍처 — 기능 간 협력 구조

세션 시작 CLAUDE.md 자동 로드 → SessionStart Hook 역할별 문서 주입 작업 계획 Plan Mode로 영향 분석 → Skills (brainstorming/plan) 활성화 병렬 개발 Agents (dev/design/devops/qa) → 독립된 Git Worktrees에서 동시 작업 품질 게이트 PreToolUse Hook → verify_local.sh → QA 에이전트 검증 PR 통합 Skills (commit/review-pr) → master 차단 Hook → 오케스트레이터 승인 머지 자동 동기화 PostToolUse Hook → 전체 워크트리 master 동기화 → Discord 알림
