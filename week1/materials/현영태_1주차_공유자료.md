## 1. CLAUDE.md — 프로젝트 기억 시스템

### 개념

**CLAUDE.md**는 세션이 시작될 때마다 Claude의 컨텍스트 윈도우에 자동으로 로드되는 마크다운 파일이다.  
"Claude에게 이 프로젝트에 대해 항상 알고 있어야 할 것"을 적어두는 **영구 지침서**다.

> 핵심: 매 세션마다 새로 시작하는 Claude에게 "이 프로젝트가 뭔지, 어떻게 일해야 하는지"를 자동으로 알려준다.

### 파일 위치와 우선순위

```
우선순위: 더 구체적인 위치 > 더 광범위한 위치

/etc/claude-code/CLAUDE.md           ← 관리형 (조직 전체, 재정의 불가)
~/.claude/CLAUDE.md                  ← 사용자 전역 (모든 프로젝트)
./CLAUDE.md 또는 ./.claude/CLAUDE.md  ← 프로젝트 (팀 공유, git 커밋)
```

하위 디렉토리의 CLAUDE.md는 Claude가 해당 디렉토리 파일을 읽을 때 지연 로드됨.

### 실제 이 프로젝트의 CLAUDE.md 계층

```
~/.claude/CLAUDE.md
  └ 한국어 ↔ 영어 처리 지침
  └ WSL 경로 규칙 (절대 경로 사용)
  └ 워크트리 구조 문서화

MustGoOutNow/CLAUDE.md               ← 오케스트레이터
  └ 역할 및 책임 정의
  └ 서브에이전트 위임 구조
  └ Git 브랜치 & PR 워크플로우

MustGoOutNow_dev/CLAUDE.md           ← Dev 워크트리
MustGoOutNow_design/CLAUDE.md        ← Design 워크트리
MustGoOutNow_devops/CLAUDE.md        ← DevOps 워크트리
MustGoOutNow_qa/CLAUDE.md            ← QA 워크트리
```

### .claude/rules/ — 경로별 조건부 규칙

```markdown
---
paths:
  - "lib/**/*.dart"      ← 이 파일들을 읽을 때만 로드됨
---

# Flutter 코드 규칙
- flutter analyze 0 Errors 유지 (zero-warning policy)
- 상태관리: Riverpod 사용
```

### 효과적인 CLAUDE.md 작성법

| 나쁜 예 | 좋은 예 |
|--------|--------|
| "코드를 잘 포맷합니다" | "2칸 들여쓰기 사용, Dart fmt 준수" |
| "변경사항 테스트하기" | "커밋 전 `flutter analyze && flutter test` 실행" |
| "파일 정리" | "서비스 클래스는 `lib/services/`에 위치" |

권장 크기: **200줄 이하** (초과 시 컨텍스트 소비 증가, 준수율 저하)

## 2. Hooks — 라이프사이클 자동화

### 개념

**Hooks**는 Claude Code의 수명 주기 중 특정 이벤트 발생 시 **자동으로 실행**되는 외부 명령이다.  
Claude에게 "해줘"라고 부탁하는 것이 아니라 **시스템 수준에서 강제**한다.

### Hook 이벤트 전체 목록

```
SessionStart          ← 세션 시작·재개 시
SessionEnd            ← 세션 종료 시
UserPromptSubmit      ← 사용자 입력 제출 직후 (Claude 처리 전)
PreToolUse            ← 도구 실행 전 (차단 가능)
PostToolUse           ← 도구 실행 후
PostToolUseFailure    ← 도구 실패 시
PermissionRequest     ← 권한 대화상자 표시 전
PermissionDenied      ← 자동 모드에서 도구 거부 시
SubagentStart         ← 서브에이전트 시작 시
SubagentStop          ← 서브에이전트 완료 시
TaskCreated           ← 작업 생성 시
TaskCompleted         ← 작업 완료 시
FileChanged           ← 감시 파일 변경 시
CwdChanged            ← 작업 디렉토리 변경 시
Notification          ← Claude가 주의를 요청할 때
```

### Hook 타입 4가지

```json
// ① Command - 셸 명령 실행 (가장 일반적)
{ "type": "command", "command": "./validate.sh", "timeout": 30 }

// ② HTTP - 외부 서비스 호출
{ "type": "http", "url": "http://localhost:8080/hooks/pre-tool", 
  "headers": { "Authorization": "Bearer $TOKEN" } }

// ③ Prompt - LLM이 직접 판단
{ "type": "prompt", "prompt": "Should this command run? $ARGUMENTS", "model": "haiku" }

// ④ Agent - 서브에이전트로 검증
{ "type": "agent", "prompt": "Verify this output: $ARGUMENTS" }
```

### Hook 제어 신호

| 종료 코드 | 의미 | 동작 |
|----------|------|------|
| `0` | 성공 | JSON stdout 처리 후 계속 |
| `2` | **차단** | stderr 메시지를 Claude에게 피드백, 중단 |
| 기타 | 경고 | 로그만 남기고 계속 |

```python
# Python hook 차단 예시
import json, sys

data = json.loads(sys.stdin.read())
command = data.get("tool_input", {}).get("command", "")

if "rm -rf" in command:
    print(json.dumps({"decision": "block", "reason": "위험한 명령 차단"}))
    sys.exit(2)

sys.exit(0)  # 허용
```

### 실제 이 프로젝트의 Hooks

#### ① `SessionStart` — 역할별 컨텍스트 자동 주입
```python
# hook_session_start.py
# cwd명 또는 subagent_type으로 역할 감지
# General/ 폴더의 최신 기획 문서·히스토리를 Claude 컨텍스트에 주입

ROLE_DOCS = {
    "dev":    ["dev/history", "planning/final_blueprint.md", "..."],
    "qa":     ["qa/history", "planning/test_cases.md", "..."],
    "design": ["design/history", "planning/detailed_specs_v3.md"],
    "orchestrator": ["모든 role history + planning 문서"],
}
# → stdout으로 문서 내용을 출력 → Claude가 컨텍스트로 수신
```

**효과**: 에이전트가 시작할 때마다 "어디까지 왔는지, 무슨 스펙인지"가 자동으로 주입됨

#### ② `PreToolUse` — master 직접 푸시 차단
```python
# hook_pre_git.py
# Bash 명령에서 "git push origin master/main" 패턴 감지 → 차단
# git commit/push 전 verify_local.sh 자동 실행
# (flutter pub get → dart format → flutter analyze → flutter test)

if is_direct_master_push(command) and not is_orchestrator(cwd):
    block("master/main 직접 푸시 금지. PR Flow 사용하세요.")
    # → exit(2)로 종료 → 명령 실행 안 됨
```

**효과**: Claude가 실수로 master에 직접 푸시하는 것을 시스템이 막음

#### ③ `PostToolUse` — 에이전트 완료 알림
```python
# hook_agent_notify.py
# Agent 도구 완료 후 Discord 알림 발송
# "✅ Dev 에이전트 작업 완료"
```

#### ④ `SubagentStart` — 서브에이전트 컨텍스트 주입
```json
// SessionStart와 동일 스크립트 재사용
// → 서브에이전트 시작 시도 역할에 맞는 문서 주입
```

## 3. Skills — 확장 가능한 명령 시스템

### 개념

**Skills**는 `SKILL.md` 파일로 정의하는 **재사용 가능한 프롬프트 템플릿**이다.  
`/skill-name`으로 직접 호출하거나, Claude가 상황에 맞게 자동으로 로드한다.

> 핵심 비유: "Claude에게 어떤 일을 '어떻게' 할지 가르치는 플레이북"

### SKILL.md 파일 구조

```yaml
---
name: explain-code
description: 코드를 다이어그램과 유추로 설명. "이게 어떻게 동작해?"라고 할 때 사용.
disable-model-invocation: false    # false면 Claude가 자동 호출 가능
user-invocable: true               # true면 /explain-code 메뉴에 표시
allowed-tools: Read Grep           # 이 Skill 활성 시 추가 허용 도구
context: fork                      # fork면 서브에이전트에서 격리 실행
model: haiku                       # 사용 모델 재정의
paths: "**/*.dart"                 # 이 파일 유형에서만 자동 활성화
---

# Skill 지침 (Claude가 따르는 내용)

코드를 설명할 때 반드시 포함:
1. **유추**: 일상 사물에 비유
2. **다이어그램**: ASCII 아트로 구조 시각화
3. **단계별 설명**: 순서대로 설명
4. **함정**: 흔한 오해 짚기
```

### 호출 방식 비교

| Frontmatter | 사용자 호출 (`/name`) | Claude 자동 호출 |
|-------------|---------------------|----------------|
| 기본값 | ✅ | ✅ |
| `disable-model-invocation: true` | ✅ | ❌ (명시적으로 막음) |
| `user-invocable: false` | ❌ | ✅ (내부 지식용) |

**`disable-model-invocation: true`를 언제 써야 하나?**  
→ `/deploy`, `/commit`, `/push` 같이 **부작용이 있는 명령**  
→ Claude가 "코드 준비된 것 같으니 알아서 배포" 하는 것을 막을 때

### 동적 컨텍스트 주입 (! 구문)

```yaml
---
name: pr-summary
context: fork
---

## 현재 PR 정보
- Diff: !`gh pr diff`
- 댓글: !`gh pr view --comments`
- 변경 파일: !`gh pr diff --name-only`

위 정보를 바탕으로 PR을 요약해줘.
```

- `` !`command` `` → Claude가 받기 **전에** 셸 명령 실행 → 출력이 프롬프트에 삽입
- 실시간 데이터 (git diff, env 값 등)를 Skill에 주입 가능

### Skill 저장 위치

| 위치 | 경로 | 범위 |
|------|------|------|
| Personal | `~/.claude/skills/<skill>/SKILL.md` | 모든 프로젝트 |
| Project | `.claude/skills/<skill>/SKILL.md` | 현재 프로젝트 |
| Plugin | `<plugin>/skills/<skill>/SKILL.md` | 플러그인 활성 시 |

### 번들 Skills (Claude Code 내장)

| Skill | 기능 |
|-------|------|
| `/batch <instruction>` | 대규모 변경을 병렬 에이전트로 조율 |
| `/simplify [focus]` | 최근 변경 파일 코드 품질 검토·수정 |
| `/loop [interval] <prompt>` | 주기적 작업 반복 실행 |
| `/debug [description]` | 디버그 로그 분석 |

### Superpowers 플러그인 Skills (이 프로젝트에서 사용)

```
/brainstorming              ← 창의 작업 전 요구사항·설계 탐색 (자동 트리거)
/writing-plans              ← 구현 계획 수립
/executing-plans            ← 계획 단계별 실행 + 체크포인트
/test-driven-development    ← 구현 전 테스트 먼저 작성
/systematic-debugging       ← 체계적 디버깅 절차
/verification-before-completion ← 완료 선언 전 검증
/requesting-code-review     ← 코드 리뷰 요청 절차
/receiving-code-review      ← 코드 리뷰 수용 절차
/writing-skills             ← 새 Skill 작성 가이드
/dispatching-parallel-agents ← 독립 작업 병렬 에이전트 배분
```

## 4. Agents (Sub-agents) — 전문화된 AI 어시스턴트

### 개념

**Sub-agent**는 **독립된 컨텍스트 윈도우**에서 실행되는 전문화된 AI 인스턴스다.  
주 대화가 위임하면 독립적으로 작업하고 요약 결과를 반환한다.

**주요 장점:**
- **컨텍스트 보존**: 탐색·연구 결과가 주 대화를 오염시키지 않음
- **도구 제한**: 에이전트마다 허용 도구 차별화 (보안)
- **모델 최적화**: 탐색 → Haiku(빠름), 구현 → Sonnet, 설계 → Opus
- **병렬 처리**: 독립 작업을 동시에 수행

### 내장 Sub-agents

| 에이전트 | 모델 | 도구 | 사용 시점 |
|---------|------|------|----------|
| **Explore** | Haiku | 읽기 전용 | 코드베이스 탐색, 파일 검색 |
| **Plan** | 주 대화 상속 | 읽기 전용 | Plan Mode에서 코드베이스 연구 |
| **General-purpose** | 주 대화 상속 | 모든 도구 | 복잡한 다단계 작업 |

### 커스텀 Sub-agent 정의 파일

```markdown
---
name: code-reviewer
description: 코드 품질·보안 검토 전문가. 코드 수정 후 즉시 실행.
tools: Read, Grep, Glob, Bash
model: sonnet
permissionMode: default
memory: project         ← 대화 간 학습 유지 (.claude/agent-memory/에 저장)
isolation: worktree     ← 격리된 git worktree에서 실행
background: false       ← foreground 실행 (차단)
maxTurns: 50
skills:
  - api-conventions     ← 시작 시 이 Skill 내용을 컨텍스트에 주입
hooks:
  PreToolUse:
    - matcher: "Edit|Write"
      hooks:
        - type: command
          command: "./scripts/run-linter.sh"
---

당신은 시니어 코드 리뷰어입니다.

[에이전트 시스템 프롬프트 작성...]
```

### Sub-agent 저장 위치 및 우선순위

```
1순위: 관리 설정 (조직 전체)
2순위: CLI --agents 플래그 (현재 세션)
3순위: .claude/agents/ (현재 프로젝트)
4순위: ~/.claude/agents/ (모든 프로젝트)
5순위: Plugin의 agents/ (플러그인)
```

### Sub-agent 호출 방법

```text
# 1. 자연어 (Claude가 판단)
"code-reviewer 에이전트로 인증 모듈 검토해줘"

# 2. @-mention (강제 지정)
@"code-reviewer (agent)" 최근 변경 사항 검토해줘

# 3. CLI 플래그 (전체 세션을 이 에이전트로)
claude --agent code-reviewer

# 4. 설정 파일 (프로젝트 기본 에이전트)
{ "agent": "code-reviewer" }
```

### 지속적 메모리 (Memory)

```yaml
memory: project   # .claude/agent-memory/code-reviewer/ 에 저장
                  # MEMORY.md가 자동 생성되어 다음 세션에 로드됨
```

| 범위 | 경로 | 공유 |
|------|------|------|
| `user` | `~/.claude/agent-memory/<name>/` | 모든 프로젝트 |
| `project` | `.claude/agent-memory/<name>/` | 버전 관리 가능 |
| `local` | `.claude/agent-memory-local/<name>/` | gitignored |

### 실제 이 프로젝트의 에이전트 구조

```
.claude/agents/
├── dev.md        → TimelineEngine, 상태관리, DB (Sonnet, MustGoOutNow_dev/)
├── design.md     → UI/UX, 타임라인 시각화 (Sonnet, MustGoOutNow_design/)
├── devops.md     → CI/CD, GitHub Actions (Sonnet, MustGoOutNow_devops/)
├── qa.md         → flutter test, 코드-스펙 정합성 (Sonnet, MustGoOutNow_qa/)
└── orchestrator.md → 전체 조율 (Opus, MustGoOutNow/)
```

**오케스트레이터 → 에이전트 위임 예시:**
```
오케스트레이터 (Opus)
  └→ dev 에이전트에게 위임: "TimelineEngine Push-back 알고리즘 구현"
  └→ qa 에이전트에게 위임: "구현된 알고리즘 테스트 검증"
  └→ 결과 통합 후 PR 머지 결정
```

## 5. Plugins — 기능 패키징 및 배포

### 개념

**Plugin**은 Skills, Agents, Hooks, MCP 서버를 하나의 패키지로 묶어 **공유·배포**할 수 있는 단위다.  
독립 실행형 `.claude/` 설정과의 차이는 **재사용성과 버전 관리**에 있다.

### 독립 실행형 vs 플러그인

| | 독립 실행형 (`.claude/`) | 플러그인 |
|--|---|---|
| **Skill 이름** | `/hello` | `/my-plugin:hello` |
| **공유** | 수동 복사 | 마켓플레이스 설치 |
| **버전 관리** | 없음 | Semantic Versioning |
| **적합한 경우** | 단일 프로젝트 | 팀·커뮤니티 배포 |

### 플러그인 구조

```
my-plugin/
├── .claude-plugin/
│   └── plugin.json        ← 플러그인 매니페스트 (이름, 버전, 설명)
├── skills/                ← Skill 정의
│   └── code-review/
│       └── SKILL.md
├── agents/                ← Agent 정의
│   └── reviewer.md
├── hooks/                 ← Hook 설정
│   └── hooks.json
├── .mcp.json              ← MCP 서버 설정
└── settings.json          ← 플러그인 기본 설정
```

```json
// .claude-plugin/plugin.json
{
  "name": "my-plugin",
  "description": "팀 공유용 코드 리뷰 플러그인",
  "version": "1.0.0",
  "author": { "name": "팀명" }
}
```

### 플러그인 설치 방법

```bash
# 마켓플레이스에서 설치
/plugin install superpowers

# 로컬 테스트
claude --plugin-dir ./my-plugin

# 변경 후 리로드 (재시작 불필요)
/reload-plugins
```

### 이 프로젝트에서 사용 중인 플러그인

```json
// ~/.claude/plugins/installed_plugins.json
{
  "plugins": {
    "superpowers@claude-plugins-official": [
      { "version": "5.0.7", "scope": "user" }
    ],
    "frontend-design@claude-plugins-official": [
      { "scope": "project" }
    ],
    "code-review@claude-plugins-official": [
      { "scope": "project" }
    ]
  }
}
```

| 플러그인 | 제공 기능 |
|---------|---------|
| **superpowers** | brainstorming, TDD, debugging, plan 등 워크플로우 Skills |
| **frontend-design** | 고품질 UI/UX 컴포넌트 생성 Skill |
| **code-review** | PR 코드 리뷰 전문 Agent |

## 6. Git Worktrees — 병렬 세션 격리

### 개념

**Git Worktree**는 동일한 저장소에서 **여러 작업 디렉토리를 동시에 체크아웃**하는 Git 기능이다.  
각 Claude 세션이 독립된 파일 공간에서 작업 → 충돌 없는 병렬 개발 가능.

```
하나의 git 저장소 (공유 히스토리·원격)
    ├── MustGoOutNow/       ← main 브랜치 (오케스트레이터)
    ├── MustGoOutNow_dev/   ← role/dev 브랜치 (Dev 에이전트)
    ├── MustGoOutNow_design/ ← role/design 브랜치 (Design 에이전트)
    ├── MustGoOutNow_devops/ ← role/devops 브랜치 (DevOps 에이전트)
    └── MustGoOutNow_qa/    ← role/qa 브랜치 (QA 에이전트)
```

### Worktree 사용법

```bash
# Claude --worktree 플래그로 자동 생성
claude --worktree feature-auth         # .claude/worktrees/feature-auth/ 생성

# 수동으로 생성 (기존 방식)
git worktree add ../project-feature-a -b feature-a
cd ../project-feature-a && claude

# 목록 확인 및 정리
git worktree list
git worktree remove ../project-feature-a
```

### Subagent Worktree 격리

```yaml
---
name: dev
isolation: worktree    # 이 에이전트 실행 시 별도 worktree 생성
---
```
→ 변경 사항 없으면 자동 정리 / 변경 있으면 유지 여부 질문

### 이 프로젝트의 워크트리 협업 흐름

```
1. 오케스트레이터 → dev 에이전트에 기능 구현 위임
2. dev 에이전트 → MustGoOutNow_dev/ (role/dev 브랜치)에서 작업
3. qa 에이전트 → MustGoOutNow_qa/ (role/qa 브랜치)에서 동시에 테스트 준비
4. 각자 PR 생성 → 오케스트레이터가 머지 결정
   (hook_post_pr_merge.py → 전체 워크트리 자동 동기화)
```

### sync_worktrees.sh — PR 머지 후 자동 동기화

```bash
# gh pr merge 실행 시 hook이 자동으로 호출:
# 1. local master pull
# 2. 전체 role 워크트리를 master로 reset
# 수동 실행: bash General/scripts/sync_worktrees.sh
```

## 7. Plan Mode — 안전한 읽기 전용 분석

### 개념

**Plan Mode**는 Claude가 **파일을 수정하거나 명령을 실행하지 않고** 코드베이스를 분석하고 계획을 세우는 모드다.

> "무엇을 할지 먼저 합의한 다음, 실행한다"

### 활성화 방법

```bash
# CLI 플래그로 시작
claude --permission-mode plan

# 세션 중 토글
Shift+Tab → Shift+Tab  (Normal → AutoAccept → Plan Mode)

# 설정 파일로 기본값 설정
{ "permissions": { "defaultMode": "plan" } }
```

### Plan Mode에서 할 수 있는 것

| 가능 | 불가능 |
|------|--------|
| 파일 읽기 | 파일 쓰기/편집 |
| Grep, Glob 검색 | Bash 명령 실행 |
| 분석 및 계획 제안 | 코드 변경 |
| 질문을 통한 요구사항 수집 | PR 생성 |

### 이 프로젝트에서의 활용

```
Superpowers /writing-plans Skill
  → Plan Mode와 연계하여 구현 계획 수립
  → 사용자 승인 후 /executing-plans로 단계별 실행
```

## 9. 실제 프로젝트 적용 총정리

### 전체 기술 스택

| 기능 | 이 프로젝트에서의 사용 |
|------|-------------------|
| **CLAUDE.md** | 5개 계층 (홈·General·4개 워크트리), 역할별 지침 분리 |
| **Settings** | 5개 계층 (전역/로컬/프로젝트/워크트리/local), 워크트리별 Write 권한 분리 |
| **Permissions** | Agent() 허용, WebFetch 도메인 제한, Bash 명령 화이트리스트 |
| **Hooks** | SessionStart(컨텍스트 주입) + PreToolUse(git 보호) + PostToolUse(알림) |
| **Skills** | Superpowers 플러그인 10개 Skills로 워크플로우 표준화 |
| **Agents** | 4개 역할 에이전트 (dev/design/devops/qa) + 오케스트레이터 |
| **Plugins** | superpowers + frontend-design + code-review 3개 |
| **Git Worktrees** | 5개 워크트리 (master + 4개 role 브랜치) 동시 운영 |
| **Plan Mode** | /writing-plans Skill과 연계한 계획 수립 |
| **Headless Mode** | verify_local.sh 내부에서 활용 |
| **GitHub Actions** | test→build_android→build_web→notify 파이프라인 |
| **Auto Memory** | 각 에이전트의 세션 간 학습 지속 |

### 협업 워크플로우 전체 흐름

```
1. 사용자 → 오케스트레이터에게 기능 요청
2. /brainstorming → 요구사항 탐색
3. /writing-plans → 구현 계획 수립 (Plan Mode)
4. 오케스트레이터 → dev 에이전트에 위임 (Agent tool)
   - SessionStart hook → dev 관련 문서 자동 주입
   - MustGoOutNow_dev/ 워크트리에서 작업
5. dev 에이전트 → 코드 구현
   - /tdd Skill → 테스트 먼저 작성
   - git commit 시 PreToolUse hook → verify_local.sh 자동 실행
   - 검증 실패 → 차단, 수정 후 재시도
6. dev 에이전트 → git push origin role/dev → gh pr create
7. PostToolUse hook → Discord 알림 발송
8. qa 에이전트 → PR 검증 (병렬로 QA 워크트리에서)
9. Gemini Reviewer 검수 (General/PR_Review/)
10. 오케스트레이터 → gh pr merge
    - hook_post_pr_merge.py → 전체 워크트리 자동 동기화
11. GitHub Actions → CI 파이프라인 실행
    - flutter test → build → GitHub Pages 배포
    - ntfy.sh → 디바이스 알림
```
