# Why Eval Matters in Harness Engineering
## Agentic Systems, Gradual Autonomy, and the Human Role

---

## 1. 문제 제기: 왜 지금 Eval인가? (운영 시스템 관점)

전통적인 소프트웨어 시스템에서 실패는 보통 크고 명확했다.

- 서비스가 중단된다
- 에러 로그와 알림이 발생한다
- 사람이 원인을 분석하고 대응한다

하지만 AI Agent 기반 시스템의 실패는 다르다.

---

### Agent는 조용히 실패한다

예를 들어, Cloud / Infrastructure Agent를 생각해보자.

- Terraform plan 생성 ✅
- Apply 실행 ✅
- 성공 로그 ✅

하지만 실제로는:
- 잘못된 리소스를 수정했거나
- blast radius가 과도했거나
- rollback 전략이 없거나
- 비용만 증가했을 수도 있다

👉 결과는 “성공”처럼 보이지만, 
판단은 실패였을 수 있다.

문제의 원인은 모델 성능이 아니라, 
“이 Agent에게 이 판단을 맡겨도 되었는가?”를 
사전에 검증하지 않았다는 점이다.

이 지점에서 Eval Harness가 필요해진다.

---

## 2. Anthropic 관점: Eval은 성능 측정이 아니다

Anthropic의 핵심 메시지는 단순하다.

Eval은 QA도 아니고, 벤치마크도 아니다. 
Eval은 “이 Agent에게 자율성을 줘도 되는가”를 판단하는 장치다.

---

### 2.1 Agent는 “출력”이 아니라 “행동”을 한다

Anthropic이 정의하는 Agent는:

- 여러 turn에 걸쳐
- 계획을 세우고
- 도구(tool)를 호출하고
- 환경(state)을 변경하며
- 이전 행동의 결과에 적응하는 시스템이다

따라서 Eval의 대상은:
- 단일 응답 ❌
- 행동 전체의 궤적(trajectory) ✅

Agent Eval은 결과보다 판단 과정을 본다.

---

### 2.2 기존 LLM Eval이 Agent에 맞지 않는 이유

기존 LLM 평가는 주로 다음을 본다.

- 정답 여부
- 출력 품질
- 문장 유사도

하지만 Agent에서는 핵심을 놓친다.

- 중간 판단이 합리적이었는가
- 올바른 tool을 선택했는가
- 실패가 다음 판단을 오염시키지 않았는가
- 불확실할 때 멈추거나 질문했는가

한 번의 잘못된 판단이 이후 모든 행동을 망칠 수 있다.

---

### 2.3 Anthropic이 말하는 Agent Eval의 구성

Agent Eval은 보통 다음 세 단위로 구성된다.

#### (1) Task
- 단순 질문이 아님
- 환경, 제약, 성공 조건을 포함한 시나리오
- 실제 운영과 구조적으로 유사해야 함

#### (2) Trial
- 동일 Task를 여러 번 실행
- 비결정성(stochastic behavior)을 전제로 평가
- “한 번 성공”은 신뢰의 근거가 아님

#### (3) Grader
- 정답 채점이 아님
- 다음을 종합 평가
  - 최종 결과
  - 중간 행동의 타당성
  - tool 선택과 순서
  - 실패 후 회복
  - 멈춤 / 질문 / 에스컬레이션 여부

---

### 2.4 가장 중요한 평가 능력: 언제 멈출 줄 아는가

불확실할 때 멈추거나 인간에게 요청하는 행동은 실패가 아니라 고급 능력이다.

Eval은 다음을 본다.

- 언제 자신 있게 진행하는가
- 언제 권한을 반납하는가

이때 Eval은 자연스럽게 Autonomy Gate 역할을 한다.

---

## 3. 핵심 용어 정리

### Eval Harness
Agent를 둘러싼 평가 인프라 전체.

- 행동 기록
- 시나리오 기반 반복 실행
- 평가 로직(Grader)

인간의 판단을 코드와 구조로 고정한다.

---

### Autonomy Gate
Eval 결과를 바탕으로 다음 중 하나를 결정한다.

- 자동 실행
- 인간 승인
- 차단 / 에스컬레이션

Eval이 실제 권한으로 연결되는 지점이다.

---

### Granularity
Human이 얼마나 자주, 어느 강도로 개입하는가를 조절하는 축.

- 모든 step 승인
- 고위험 행동만 개입
- 예외 상황만 호출
- 사후 감사만 수행

Human‑in‑the‑loop은 on/off가 아니라 연속 조절 문제다.

---

### Gradual Autonomy
자율성을 한 번에 주지 않고 
Eval → Score → Gate를 근거로 단계적으로 확대하는 전략.

---

## 4. 그러면 Score는 어떻게 측정하는가?

Agent Eval의 Score는 하나의 숫자가 아니라 
여러 판단을 종합한 자율성 신뢰도 신호다.

Score는 다차원 평가로 구성된다.

---

### 4.1 Score를 구성하는 대표적인 평가 축

#### ① Goal Achievement
- 목표 조건을 만족했는가
- 완전 성공 / 부분 성공 / 실패

#### ② Action Correctness
- 올바른 tool 선택
- 파라미터 정확성
- 불필요하거나 위험한 행동 여부

#### ③ Trajectory Quality
- 불필요한 step 최소화
- 무한 루프 방지
- 실패 후 회복 능력

#### ④ Uncertainty Handling
- 정보 부족 인식
- 실행 중단
- Human 에스컬레이션

멈춤은 고급 능력이다.

#### ⑤ Safety & Policy Compliance
- 권한 없는 행동
- 정책 위반
- 보안 / 비용 가드레일 침범

보통 fail-fast 조건으로 처리된다.

#### ⑥ Cost / Efficiency
- step 수
- tool call 수
- latency
- 토큰 및 비용

---

### 4.2 Score를 숫자로 만드는 방식

#### 방식 1: 가중 합산
```
Final Score =
  0.25 * Goal Achievement
+ 0.20 * Action Correctness
+ 0.20 * Trajectory Quality
+ 0.20 * Uncertainty Handling
+ 0.10 * Safety
+ 0.05 * Cost
```

#### 방식 2: Rule + Score 혼합
```
if safety_violation:
    block_action
elif final_score >= 0.95:
    auto_execute
elif final_score >= 0.80:
    require_human_approval
else:
    escalate_to_human
```

Score는 결론이 아니라 Autonomy Gate의 입력값이다.

---

### 4.3 Human은 Score의 어느 지점에 있는가?

Human의 역할은 다음과 같다.

- 평가 축 정의
- 가중치 설정
- threshold 결정
- 모호한 케이스 보정

Human은 점수를 매기지 않는다. 
점수가 의미하는 바를 정의한다.

---

## 5. Human 역할 변화와 Score의 관계

### 조작자 → 승인자 → 감독자 → 책임자

| 단계 | Human 역할 | Score의 쓰임 |
| --- | --- | --- |
| 조작자 | 직접 실행 | 기준 수집 |
| 승인자 | Yes / No | threshold 조정 |
| 감독자 | 예외 개입 | Gate 판단 |
| 책임자 | 사후 책임 | 정책 소유 |

Score는 이 역할 이동을 가능하게 하는 핵심 매개체다.

---

## 6. 핵심 한 문장 요약

Agent Eval의 Score는 성능 점수가 아니다. 
“이 판단을 맡겨도 되는가?”에 대한 
시스템의 신뢰도 응답이다.
