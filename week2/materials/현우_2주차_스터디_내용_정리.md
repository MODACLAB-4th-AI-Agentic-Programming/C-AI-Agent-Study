# 인호
- 페이퍼 클립 소개 > ai 에이전트로 회사를 만들다
- 회사처럼 굴리는 ai 조직을 만들어서 사용해보는 느낌으로
- 각각 에이전트들이 자기 일을 잘 할 수 있도록 잘 설명해주는 게 중요
- CEO에게 일감을 잘 나눠 주라고 설명해주는 방식으로 동작
- 페이퍼 클립?
	- ai 가상 회사 플랫폼
	- 각각에게 CEO, CTO 처럼 역할을 적용해서 실제 조직처럼 운영
	- 이슈/티켓 기반으로 작업 관리 > 에이전트가 자율적으로 분석, 실행, 완료
	- 서브 티켓도 자동으로 생성되며 에이전트 간 협업 가능
	- 오픈소스, 로컬 환경에서 실행
	- 꼭 클로드 코드가 아니여도 가능 (코덱스 등 연동 가능)
	- 다 하고 나서는 커밋, 푸시까지 자동으로 동작
- 4단계면 회사가 생김
	- 조직 이름(Company Name)을 붙이고 사용 가능 — 프로젝트가 아니라 "조직" 단위로 생각하는 게 포인트 (예: 로스트아크가 아니라 스마일게이트)
	- 미션과 골을 입력
	- 에이전트들을 선택할 수 있도록 해주는 (추천: Claude Code, Codex / Gemini, Sierra 등도 가능)
	- CEO는 똑똑해야 한다고 판단 → Opus 4.6 사용, 나머지는 Sonnet으로 비용 최적화
	- CEO가 엔지니어들을 고용하는 것이 첫번째 플랜
	- 실제 조직도를 볼 수 있고 어떤 일을 하는지도 볼 수 있음
	- CTO가 코드 구현 > QA 엔지니어가 받아서 버그 리포트 > CTO가 수정
	- UX 디자이너, PM 등에게 디자인 토큰·와이어프레임(ASCII)·PRD 작성 위임
	- 문서화가 안되는 경우를 대비해 컨벤션 관리까지 담당하는 Secretary(비서) 역할 추가
	- 실시간으로 작업하고 있는 것들을 바로 볼 수 있음
	- 지금은 다 CEO 밑으로 물려 있음 > CEO에게 각자 역할 잘하는지 확인하도록
- 만든 앱 "우롱" (우렁각시 컨셉, 가사 노동 공유 앱)
	- MVP는 거의 하루 만에 완성 (실제 작업한 날은 이틀)
	- 현우가 작업 중 신경 쓴 건 폴더 설정 (워킹 디렉토리 config 수정 필요)
	- 조직도 Import/Export 가능 → 다른 곳에서도 동일 에이전트 재사용
- 좋았던 점
	- 병렬 작업 가능, 문서화 자동
	- 프로세스가 점점 좋아지는 게 눈에 보임
	- 체크아웃 기반이라 충돌 없이 병렬 처리
	- 에이전트들이 이슈 받을 때만 활성화 (비용 절감)
- 아쉬운 점
	- 하트비트 단위로 동작하여 맥락이 가끔 끊김 (워킹 디렉토리 오염 이슈)
	- 블로킹 이슈(예: Firebase 설정 주입) 있으면 멈춤
	- 에이전트 간 소통 비용(토큰) 많이 듦, 빠른 편은 아님
	- 할루시네이션 자체 검토 불가 → 결국 대표(사용자)가 결과 보고 판단·수정
	- 완벽하지 않아서 방향 잡아주고 혼내가면서 진행해야 함
- 코드 품질
	- 구조와 컨벤션은 잘 지켜짐, 페이지 분리도 깔끔
	- MVP 올릴 때 한 번 이슈 있어서 수정 필요했음
- 사용처: 사이드 프로젝트, 스타트업 초기 MVP에 적합
- 발표 PPT·대본도 페이퍼 클립 내부 UX 디자이너·Secretary가 만들어줌

# 현우
- 발표 주제: 하네스(Harness) 개념을 개인 작업 플로우에 접목
- 배경
	- 랭체인 조사 자료 기준, 모델 그대로 두고 하네스만 조정해도 성능 13.7% 상승
	- 하네스 구조: Context / Suggest / Block / Loop / Evaluate
		- 비유: Context=레시피·재고 확인, Suggest=일반 조리법, Block=가스 안전장치(위험 차단), Loop=간 맞을 때까지 반복, Evaluate=마지막 평가
- 개인 작업 맥락
	- 작년 8월 입사 이후 커서로 개발 시작 — 세션 단위라 컨텍스트 유지가 어려움
	- 작업마다 **문서화를 적극적으로 쌓아둠** (Obsidian 기반)
	- 3종 문서 구조: `findings` / `progress` / `task-plan`
		- Andrej Karpathy가 공유한 아이디어에서 착안
- 시도한 것
	- **로컬 Obsidian MCP 직접 구현**: Obsidian vault(마크다운 덩어리)에서 자료 찾아주는 MCP
	- **Knowledge Agent**: MCP + CLI + 스킬 형태로 구성, 자료 찾아서 요약 답변
	- Context7이 공식문서 기반이라면, 이건 내 개인 지식을 벡터 임베딩해서 찾아주는 개인용 Context7
	- 동작 예: "market.json이 어떻게 생성되는지?" → Knowledge Agent가 MCP 통해 Obsidian에서 검색 → 요약
- 워크플로우
	- 지라 티켓 분석 에이전트가 티켓 내용 분석
	- `memory/tasks/`에 `progress`, `findings`, `task-plan` 파일 자동 생성
	- 대화로 문서 완성 후 코드 작업
	- 세션 끊겨도 문서로 복구 가능
- 느낀 점
	- 꾸준한 문서화 자산이 그대로 재사용 가능해서 뿌듯
	- 하네스는 **기본 데이터(문서)**가 있어야 구성이 쉬움
	- 클로드 팀모드(서브에이전트) 시도 → **비효율적** (한 번에 토큰 30% 소모)
	- Opus 4.7부터 컨텍스트·성능 향상 → 팀 모드로 쪼갤 필요 없다는 동료 조언 공감
- 클로드 vs 커서 비교
	- 커서는 파일 인덱싱 기반, 클로드는 전수조사식으로 읽음 → 개인 자료 검색이 느림
	- MCP로 보완하면 속도·정확도 향상
- Q&A 정리
	- **태스크 플랜 파이프라인**: 스킬로 `findings/progress/task-plan` 3종 문서 작성 자동화
	- 문서 토큰 비용 vs 효과: 특정 지면은 코드로 알 수 없는 히스토리가 많아 컨텍스트 문서가 필수 (기획자 요청 이력·판단 사유까지 기록)
	- **모순·할루시네이션**: 문서 기반이면 모순이 줄지만, 새로운 내용 질문 시 문서에서 엉뚱한 걸 끄집어내는 케이스 존재 — 개선 필요
	- 세션 복구 장점 크다는 경험 공유
- 향후 계획(야근으로 못한 것)
	- 지라 티켓 기반 전체 작업 자동화
	- 주간 작업 분석 자동화

	- 자동 수정 + 리포트 + 커밋 직전까지 안내
- 하네스100 코드 리뷰어
	- 에이전트(누가) / 스킬(지식·규칙) / CLAUDE.md(어떻게 협업) 3단 구조
	- 에이전트: 아키텍처, 퍼포먼스, 전체, 보안, 스타일 등 역할별 분리
	- 예시 — 아키텍처 리뷰어
		- SOLID 원칙 검증, 설계 패턴·의존성 분석
		- 체크리스트 검증 → 산출물 포맷 → 팀 통신 프로토콜 → 에러 처리 순
	- 스킬: 코드 스타일 가이드, 참고 리소스
	- Workflow: 팀 통신·산출물 정리·테스트 시나리오
	- CLAUDE.md: 가장 큰 설명서 — 클로드가 명령 받으면 CLAUDE.md → 스킬 순으로 탐색
	- 결과
		- `/simplify`가 놓치는 보안·아키텍처 이슈를 추가로 찾아줌 (예: 디버그 코드 잔존, 죽은 코드, 시큐리티 측면)
		- 제가 MD에 안 적은 영역까지 알아서 점검
		- **주의**: 설치 시 기존 CLAUDE.md가 덮어씌워질 수 있음 → 스냅샷 찍고 하나씩 옮기는 것 추천
	- 사용량: 하루 **27%** 소모 (기존 7% 대비 +20%)
	- 결론: 효과는 좋지만 **급한 스프린트 중엔 부담** → 바로 뺌

	- 플랜 짜기를 상당히 잘해서 플랜 기반 구현도 좋음

# 정호
- 주제: 하네스 엔지니어링 — 평가(Eval) / 피드백 체계 / 휴먼 리뷰 지점
- 배경
	- 회사 Q&A 채널에서 질문 답변을 담당
	- 70% 이상은 매크로 답변 가능 → 자동답변 시스템을 만들고 싶었음
	- Claude Code로 만들었더니 **카테고라이징이 잘 안 됨**
		- 모수가 적어서 묶이지 말아야 할 게 묶이거나
		- 답변이 잘못되는 경우 발생
	- Claude 에이전트는 자동 처리 중 **어디서 잘못됐는지 리뷰할 인터랙션이 부족**
	- → 사람이 감시·평가하고, 평가가 충분히 올라가면 다시 에이전트에 위임하는 **하네스 Eval** 개념 적용
- 적용
	- 지난주 공유한 개념 MD 파일만 프로젝트에 넣고 "이 개념 적용해줘"로 간단히 시도
	- 기존: 메시지 파싱 → Claude Opus가 깜깜이로 FAQ 분류
	- 적용 후
		- 스코어링 기법 + 어디서 잘못됐는지 점수 매김
		- 일정 스코어 미만이면 **휴먼 리뷰 단계**로 넘어와 본인이 승인/스킵
		- 카테고라이징 도움
- 다음 계획
	- 특정 Q&A에서 스코어가 계속 낮으면 스코어 계산식 재조정 or 카테고라이즈 코드 재작업
	- 척도가 될 스코어 계산을 시작점으로 고도화
- Q&A
	- 툴도 이미 나와 있어 두 개 비교해볼 예정 (회사 토큰 여유)
	- 인풋 규격화 아이디어(슬랙 양식 구조화, CLI화된 지라 티켓)도 참고
	- AI로 **페르소나를 만들어 역설문** → 가상 데이터로 테스트하는 방법도 후보

# 승현
- 주제: 멀티 에이전트 상호 리뷰 + 메모리·어드바이저 조합 시도 + RTK 후기
- 배경
	- 지난주 정우님 발표(적대적 리뷰 아이디어)가 인상적
	- Anthropic 블로그도 세션 간 상호 리뷰로 할루시네이션 감소 효과 언급
	- 메모리 / 어드바이저 (이번 Claude 업데이트 신규 기능) + 상호 리뷰 세 조합 시도 계획
	- 야근으로 풀 적용은 못 함 → 평소 쓰던 G-Stack + Superpowers 조합으로 실험
- 실험
	- G-Stack과 Superpowers가 서로의 결과를 다른 세션에서 리뷰
	- 멀티 에이전트 기대가 컸으나 **생각보다 무겁고 세션 간 정보 교환 시 누락 多**
	- 결론
		- Superpowers: **코드 레벨**에서 활용
		- G-Stack: 코드 한 칸 위 레이어(추상화·설계·PM 관점)에서 활용
		- 각자 세션을 나눠 돌리는 게 더 좋음
		- 맥락·하네스 규칙이 많아질수록 결과 제어 어려움 → **한 번에 하나씩** 진행하는 게 나음
- 앞으로 해보고 싶은 것
	- 현우가 한 "기억의 영속성" 부여를 고도화
	- 회사 프로젝트용 하네스 레이어 구축
- RTK 공유 (A조 윤수님 후기)
	- 기존: AI → CLI → 전체 출력 → AI
	- RTK 적용: RTK가 CLI 사이에서 **핵심 정보만 AI에게 전달**
	- 우려: 중요 정보 누락?
	- 실제: **삭제가 아니라 포맷 압축** → 맥락 파악 제거 체감 없음
	- 회사 사용 안전성 후기도 긍정적
	- 어제부터 직접 써보니 괜찮음
- 코덱스를 쓰는 사람들도 많음 → Claude에서 Codex를 쓰는 방법도 이번 주 시도 예정
