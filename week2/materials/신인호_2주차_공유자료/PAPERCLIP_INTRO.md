# Paperclip 소개 — AI 에이전트로 회사를 만들다

> "코드를 짜는 AI"가 아니라, "회사처럼 굴러가는 AI 조직"을 만들 수 있다면?

---

## 1. Paperclip이란?

Paperclip은 **AI 에이전트들로 구성된 가상 회사**를 만들고 운영할 수 있는 플랫폼이다.

- 각 에이전트에게 역할(CEO, CTO, 디자이너 등)을 부여
- 이슈를 만들고 할당하면, 에이전트가 자율적으로 작업 수행
- 에이전트 간 위임, 리뷰, 에스컬레이션이 실제 조직처럼 동작
- 로컬 환경(Claude Code, Codex 등)에서 코드를 직접 수정

핵심 컨셉: **사람이 보드(이사회)**가 되고, AI 에이전트들이 **실무 조직**이 된다.

---

## 2. 우리가 만든 것: ulong

**ulong**은 가족/룸메이트 간 가사 노동을 기록하고, 완료 시 상대에게 푸시 알림과 칭찬을 보내는 앱이다.

| 항목 | 내용 |
|------|------|
| 기술 스택 | React Native + Expo SDK 55 |
| 플랫폼 | iOS / Android |
| 핵심 기능 | 가사 아이콘 등록, 푸시 알림, 칭찬 리액션 |
| 현재 상태 | MVP 완성, EAS Build 설정 완료 |

이 앱을 **Paperclip 에이전트 조직이 처음부터 끝까지** 만들었다.

## 3. 에이전트 조직 구성

총 **6명의 AI 에이전트**가 각자의 역할을 수행한다.

```
        [보드 (사람)]
             |
           [CEO]
          /  |  \  \   \
       CTO  PM  UX  QA  Secretary
```

| 에이전트 | 역할 | 하는 일 |
|----------|------|---------|
| **CEO** | 경영 총괄 | 우선순위 결정, 작업 위임, 크로스팀 조율, 에이전트 채용 |
| **CTO** | 기술 총괄 | 코드 구현, 버그 수정, 인프라 설정, 기술 의사결정 |
| **PM** | 프로덕트 매니저 | PRD 작성, 스펙 정의, 일정 관리 |
| **UXDesigner** | 디자인 | UI 와이어프레임, 디자인 토큰, UX 스펙 |
| **QAEngineer** | 품질 보증 | 코드 리뷰, 테스트 환경 구성, 버그 리포트 |
| **Secretary** | 서기 | 문서화, 컨벤션 관리, 프로세스 정리 |

---

## 4. 실제 작업 흐름 — 이렇게 돌아간다

### 4.1 이슈 기반 작업

모든 작업은 **이슈(티켓)**로 관리된다.

```
보드가 이슈 생성 → CEO가 분석/위임 → 담당 에이전트가 체크아웃 → 작업 → 상태 업데이트 → 완료
```

실제 예시:
- `ULO-9` "mvp를 만들기 위한 agent 고용" → CEO가 CTO, PM 등을 순차 채용
- `ULO-16` "ulong MVP React Native 구현" → CTO가 체크아웃하여 전체 앱 구현
- `ULO-18` "MVP 코드 리뷰 및 QA" → QAEngineer가 리뷰 후 버그 목록 작성
- `ULO-19` "QA 발견 버그 수정" → CTO가 버그 수정

### 4.2 크로스팀 협업

하나의 기능이 여러 에이전트를 거쳐 완성된다:

1. **PM**이 PRD/스펙 작성 → `ULO-10`
2. **UXDesigner**가 UI 스펙 작성 → `ULO-15`
3. **CTO**가 구현 → `ULO-16`
4. **QAEngineer**가 검증 → `ULO-18`
5. **Secretary**가 문서화 → `ULO-28`

### 4.3 프로세스 자동 구축

에이전트들이 자체적으로 프로세스를 만들기도 했다:
- `ULO-32` CTO가 QA 리뷰 프로세스 도입
- `ULO-35` Secretary가 크로스팀 프로세스 문서화
- `ULO-37` CTO가 Jest 테스트 환경 구성

## 6. 좋았던 점

### 병렬 작업이 가능하다
CTO가 코드를 짜는 동안 PM은 스펙을 쓰고, UXDesigner는 와이어프레임을 그린다. 사람 한 명이 순차적으로 해야 할 일을 동시에 처리할 수 있다.

### 문서화가 자동으로 쌓인다
이슈마다 코멘트, 상태 변경, 작업 기록이 남는다. Secretary가 컨벤션 문서와 일지를 자동으로 관리해준다. 별도로 문서 작업을 하지 않아도 프로젝트 히스토리가 쌓인다.

### 프로세스가 스스로 진화한다
QA 프로세스, 코딩 컨벤션, 디자인 토큰 시스템 등을 에이전트들이 필요에 따라 자체적으로 만들고 개선했다.

### 보드(사람)는 의사결정에만 집중할 수 있다
"이거 만들어줘"라고 이슈 하나 만들면, 나머지는 에이전트 조직이 알아서 분해하고 실행한다. 사람은 방향만 잡으면 된다.

### 채용이 즉시 가능하다
새로운 역할이 필요하면 에이전트를 바로 생성할 수 있다. QA가 필요해서 `ULO-17`로 QAEngineer를 고용했고, 문서 관리가 필요해서 `ULO-27`로 Secretary를 고용했다.

## 7. 아쉬웠던 점

### 컨텍스트 유실
에이전트는 하트비트(짧은 실행 윈도우) 단위로 동작한다. 이전 하트비트의 맥락을 완벽하게 이어받지 못할 때가 있다. 복잡한 작업일수록 컨텍스트 재구성에 시간이 걸린다.

### 블로킹 이슈 해결이 느리다
에이전트가 스스로 해결하지 못하는 블로킹(예: 외부 서비스 설정, 물리 디바이스 테스트)은 사람이 개입해야 하는데, 그 사이 관련 작업들이 멈춘다.

### 에이전트 간 소통 비용
이슈 코멘트를 통한 비동기 소통은 체계적이지만, 실시간 논의가 필요한 상황에서는 비효율적이다. 각 에이전트를 깨우는 것 자체가 비용(토큰, 시간)이다.

### 디버깅이 어려울 때가 있다
에이전트가 작성한 코드에 문제가 생기면, "왜 이렇게 짰는지"를 추적하는 과정이 사람이 짠 코드보다 더 어려울 수 있다. 코멘트와 이슈 기록이 있긴 하지만 항상 충분하지는 않다.

### 로컬 환경 의존성
현재 `claude_local` 어댑터를 사용하므로 로컬 머신에서 실행된다. 리모트 환경 지원이 더 성숙해지면 이 제약은 줄어들 것이다.

## 8. 신기한 기능들

### 체크아웃 시스템
하나의 이슈를 두 에이전트가 동시에 작업하는 것을 방지한다. 마치 Git 브랜치 잠금처럼, 먼저 체크아웃한 에이전트만 작업할 수 있다. 충돌 없이 병렬 작업이 가능한 핵심 메커니즘.

### 하트비트 기반 실행
에이전트는 계속 켜져 있는 게 아니라, 이벤트(이슈 할당, 코멘트, 블로커 해제 등)가 발생하면 깨어나서 작업하고 다시 종료된다. 리소스 효율적이다.

### 자동 의존성 관리
`blockedByIssueIds`로 이슈 간 의존성을 설정하면, 블로커가 모두 해결될 때 자동으로 다음 에이전트를 깨운다. 수동으로 "A 끝났으니 B 시작해"라고 할 필요가 없다.

### 실행 정책 (Execution Policy)
이슈에 리뷰/승인 단계를 설정할 수 있다. 코드를 짜고 → 자동으로 리뷰어에게 넘어가고 → 승인하면 다음 단계로 진행. CI/CD 파이프라인과 유사한 워크플로우를 이슈 레벨에서 구현.

### 에이전트 채용 프로세스
새 에이전트를 만들 때 역할, 스킬, 지시사항을 정의하면, 해당 에이전트가 자체 온보딩을 수행한다. `ULO-1`에서 첫 엔지니어를 고용한 것이 시작점이었다.

### 이슈 문서 시스템
이슈에 `plan`, `spec` 같은 키로 문서를 첨부할 수 있다. 플랜 작성 → 리뷰 → 수정의 사이클이 이슈 안에서 완결된다.

## 9. 활용 시나리오

Paperclip이 특히 효과적인 상황:

| 시나리오 | 이유 |
|----------|------|
| **사이드 프로젝트 MVP** | 혼자서 PM/디자인/개발/QA를 다 해야 할 때, 각 역할을 에이전트에게 위임 |
| **프로토타이핑** | 아이디어 → PRD → 디자인 → 코드 → QA를 빠르게 순환 |
| **문서화가 중요한 프로젝트** | 모든 의사결정이 이슈/코멘트로 자동 기록 |
| **반복적인 유지보수** | 루틴 기능으로 정기 작업을 자동화 |

## 11. 결론

Paperclip은 "AI한테 코드 짜달라고 하는 것"과는 다른 차원의 도구다.

- AI에게 **역할**을 주고
- **조직**으로 구성하고
- **이슈**로 소통하면
- 하나의 **프로덕트**가 만들어진다

사람은 CEO(보드)로서 방향만 잡으면 된다. 나머지는 에이전트 조직이 실행한다.

아직 완벽하지는 않다. 컨텍스트 유실, 복잡한 디버깅, 로컬 환경 제약 등 개선할 점이 있다. 하지만 **"AI와 함께 일하는 방식"의 새로운 패러다임**을 보여주는 도구로서, Paperclip은 충분히 흥미롭고 실용적이다.
